1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document provides a comprehensive and formally structured description of all functional and non-functional requirements for StudentStay, a web-based student housing platform. StudentStay is designed to bridge the gap between university students seeking affordable, verified accommodation and landlords offering rental properties in proximity to educational institutions in Bangladesh.

The primary purpose of this document is to serve as a binding reference for all stakeholders — the development team, the course instructor, and potential end-users — establishing a clear and unambiguous understanding of what the system shall deliver in its initial release (Version 1.0). This document further articulates the system's scope, constraints, external interfaces, and the analysis models that underpin its architectural design.

1.4 Project Scope

StudentStay is a full-stack web application that addresses a pervasive challenge faced by university students in Bangladesh: the difficulty of finding safe, affordable, and conveniently located housing near their institutions. Traditional methods — word-of-mouth referrals, unregulated broker networks, and classified advertisements — often expose students to unreliable information, hidden fees, and substandard living conditions.

The platform provides a centralized digital marketplace where:

  • Students can search for, compare, and book verified accommodation listings filtered by university proximity, price range, property type, amenities, and gender-specific rules.
  • Landlords can publish detailed property listings, manage incoming booking requests, and build a reputation through verified tenant reviews.
  • Administrators can oversee platform integrity by managing user accounts, moderating property listings, and accessing operational analytics.

Version 1.0 includes:

  • Secure JWT-based user authentication with role differentiation (Student, Landlord, Admin)
  • Full CRUD operations for property listings with multi-criteria advanced search
  • A structured booking workflow with status lifecycle management
  • A multi-dimensional review and rating system for properties
  • Role-specific dashboards with real-time aggregate statistics
  • Responsive front-end designed for both desktop and mobile browsers

Explicitly outside scope for Version 1.0:

  • Online payment gateway integration (payment tracking is status-based only)
  • Real-time chat or messaging between students and landlords
  • Mobile native applications (iOS/Android)
  • Map-based geolocation and route visualization
  • AI-powered property recommendation engine

2. Overall Description

2.1 Product Perspective

StudentStay is a self-contained, independently developed web application and is not a component of any larger existing system. It is conceived as a modern replacement for the informal and fragmented channels through which students in Bangladeshi cities currently seek rental housing.

The system follows a three-tier client–server architecture:

  • Presentation Tier: A responsive single-page-style front-end built with HTML5, CSS3, and vanilla JavaScript, served as static files by the application server.
  • Application Tier: A RESTful API server built on Node.js with Express.js, handling business logic, authentication, authorization, input validation, and request routing.
  • Data Tier: A MongoDB NoSQL database managed via Mongoose ODM, storing all persistent data including users, properties, bookings, and reviews.

The system is designed for deployment on standard cloud infrastructure and communicates exclusively over HTTPS. Security measures include Helmet.js for HTTP header hardening, CORS policy enforcement, bcrypt-based password hashing, and API rate limiting.

2.3 User Classes and Characteristics

The system defines three distinct user roles, each with specific privileges and interaction patterns:

2.3.1 Student

Students represent the primary consumer-side users of the platform. They are typically undergraduate or postgraduate students enrolled in Bangladeshi universities who are seeking off-campus housing. Students are expected to have basic digital literacy and regular access to a web browser. Key capabilities include browsing and searching properties, submitting booking requests, writing reviews, and managing their own profile and booking history.

2.3.2 Landlord

Landlords are property owners or managers who wish to advertise and rent accommodation to students. They may range from individual homeowners renting a single room to small-scale hostel operators. Landlords are expected to provide accurate property information and respond to booking requests in a timely manner. Key capabilities include creating and managing property listings, reviewing and responding to booking requests, and monitoring their property portfolio through the dashboard.

2.3.3 Administrator

Administrators are system operators with elevated privileges. They are responsible for maintaining platform integrity, managing user accounts, moderating property listings, overseeing bookings, and accessing aggregate platform statistics. The administrator role is assigned internally and is not available through self-registration.

2.5 Design and Implementation Limitations

  • The system does not integrate with any external payment gateway in Version 1.0; payment status is tracked manually.
  • Property images are referenced via URL strings; the system does not currently include a dedicated file upload or image storage service.
  • The front-end is built with vanilla JavaScript without a reactive framework, which may limit UI complexity for future features.
  • MongoDB is used as a NoSQL database; certain relational integrity constraints must be enforced at the application layer rather than the database layer.
  • The system uses a single-server architecture; horizontal scaling and load balancing are outside the scope of Version 1.0.
  • Rate limiting is configured at 100 requests per 15-minute window per IP, which may need tuning under production traffic.

2.6 User Documentation

The following documentation resources shall accompany the system:

  • README.md: A project-level guide covering installation, configuration, and startup instructions.
  • DATABASE_SETUP.md: Detailed instructions for MongoDB setup, including both local and Atlas cloud configurations.
  • API Reference: Inline documentation of all RESTful endpoints, their expected request/response structures, and authentication requirements.
  • In-Application Guidance: Context-sensitive labels, placeholder text, and form validation messages to assist users during interaction.

2.7 Assumptions and Dependencies

Assumptions:

  • Users have access to a modern web browser and a stable internet connection.
  • Landlords will provide truthful and accurate property information.
  • Students will use the review system responsibly and in good faith.
  • The system will initially operate within the Bangladeshi geographic context, with pricing in BDT and phone number validation following local formats.
  • A single administrative user is sufficient for Version 1.0 platform management.

Dependencies:

  • MongoDB must be accessible (locally or via MongoDB Atlas) for all data persistence operations.
  • Node.js and npm must be installed on the server environment.
  • Third-party npm packages (Express, Mongoose, bcryptjs, jsonwebtoken, Helmet, etc.) must remain available and compatible.
  • The JWT_SECRET environment variable must be securely configured before deployment.

3. System Features

This chapter describes the major functional areas of the StudentStay platform. Each feature is presented with its description, priority, stimulus/response sequences, and associated functional requirements.

3.1 User Authentication and Account Management

3.1.1 Description and Priority

Priority: High. This feature provides the foundational identity and access control layer for the entire platform. It enables users to register accounts with role selection (Student or Landlord), authenticate via email and password, maintain their profile information, and securely change their passwords. All protected system operations depend on successful authentication through this module.

3.1.2 Stimulus/Response Sequences

  • Registration: A new user completes the registration form with name, email, password, role, and optional fields (phone, university, student ID). The system validates input, checks for duplicate emails, hashes the password, creates the account, issues a JWT, and redirects to the dashboard.
  • Login: An existing user submits email and password. The system verifies credentials, updates the last-login timestamp, issues a JWT, and redirects to the dashboard.
  • Profile Update: An authenticated user modifies their name, phone, university, or student ID. The system validates and persists the changes.
  • Password Change: An authenticated user submits their current password and a new password. The system verifies the current password, hashes the new one, and updates the record.

3.1.3 Functional Requirements

FR-01 The system shall allow a new user to register by providing a name (3–50 characters), a valid email address, a password (minimum 6 characters), and a role selection of either "student" or "landlord."
FR-02 The system shall reject registration attempts if the submitted email address already exists in the database, returning an appropriate error message.
FR-03 The system shall hash all user passwords using bcrypt with a salt factor of 10 before persisting them to the database.
FR-04 The system shall issue a JSON Web Token (JWT) with a 30-day expiration upon successful registration or login.
FR-05 The system shall authenticate users by verifying the submitted password against the stored bcrypt hash and return an "Invalid credentials" message upon mismatch.
FR-06 The system shall update the lastLogin timestamp for the user upon each successful login.
FR-07 The system shall allow authenticated users to update their profile fields (name, phone, university, studentId) while enforcing all existing validation rules.
FR-08 The system shall allow authenticated users to change their password by first verifying the current password, then hashing and storing the new password.
FR-09 The system shall exclude the password field from all API responses that return user data.

3.2 Property Listing and Management

3.2.1 Description and Priority

Priority: High. This feature enables landlords to create, update, and delete property listings. Each listing captures comprehensive details including title, description, property type, full address, nearest university with distance, pricing (amount, currency, billing period), room configuration, amenities, house rules (gender preference, smoking, pets, visitor policies), contact information, and images. Administrators also have management access to all listings.

3.2.2 Stimulus/Response Sequences

  • Create Listing: An authenticated landlord fills out the property form with all required fields. The system validates input, associates the listing with the landlord's user ID, persists the data, and confirms success.
  • Update Listing: A landlord selects an existing property and modifies its details. The system verifies ownership, validates changes, and persists updates.
  • Delete Listing: A landlord requests deletion of a property. The system verifies ownership and removes the record.
  • View My Properties: A landlord navigates to their dashboard to see all their listed properties with current status.

3.2.3 Functional Requirements

FR-10 The system shall allow authenticated users with the "landlord" role to create a new property listing by submitting a title, description, type (room, apartment, pg, hostel, or studio), address details, university name, distance, pricing, room counts, and contact information.
FR-11 The system shall enforce the following property constraints: title maximum 100 characters, description maximum 2,000 characters, price non-negative, and rooms totals as positive integers.
FR-12 The system shall support the following amenity options for each property: wifi, ac, parking, laundry, kitchen, gym, security, cctv, water, electricity, gas, elevator, generator, and furnished.
FR-13 The system shall allow landlords to specify house rules for each property, including gender preference (male, female, or any), smoking policy, pet policy, and visitor policy.
FR-14 The system shall allow only the owning landlord or an administrator to update or delete a property listing, returning a 403 Forbidden status for unauthorized attempts.
FR-15 The system shall assign a default status of "available" to newly created property listings.
FR-16 The system shall maintain a view counter for each property, incrementing it by one each time the property detail page is accessed.

3.3 Property Search and Discovery

3.3.1 Description and Priority

Priority: High. This feature provides students with a powerful multi-criteria search engine to discover available properties. Users can filter results by university name, city, district, property type, price range, amenities, and gender preference. A free-text search option scans across title, description, university, area, city, and district fields. Results are paginated and sortable.

3.3.2 Stimulus/Response Sequences

  • Filtered Search: A student selects one or more filter criteria on the properties page. The system constructs an aggregated query, retrieves matching available properties, and displays paginated results.
  • Free-Text Search: A student types a keyword into the search bar. The system performs a case-insensitive regex search across multiple fields and returns matching results.
  • View Property Details: A student clicks on a property card. The system retrieves full property details including landlord information and associated reviews.

3.3.3 Functional Requirements

FR-17 The system shall support property search with the following optional filter parameters: university, type, minPrice, maxPrice, city, district (zilla), amenities (comma-separated), and gender.
FR-18 The system shall support a free-text search parameter that performs case-insensitive matching across the title, university, area, city, and district fields.
FR-19 The system shall return only properties with a status of "available" in all public search and browse queries.
FR-20 The system shall paginate search results with a default of 12 properties per page and include pagination metadata (total count, current page, total pages) in the response.
FR-21 The system shall support sorting of search results, defaulting to most recently created first (-createdAt).
FR-22 The system shall populate landlord information (name, email, phone, profile image) in property detail responses.

3.4 Booking Management

3.4.1 Description and Priority

Priority: High. This feature enables students to submit booking requests for available properties and landlords to manage those requests. The booking lifecycle includes creation, confirmation, rejection, cancellation, and completion. Availability tracking is maintained by decrementing available room counts upon booking and restoring them upon cancellation or rejection.

3.4.2 Stimulus/Response Sequences

  • Create Booking: A student selects a property, specifies start/end dates and notes, and submits. The system validates availability, creates a pending booking, decrements available rooms, and confirms.
  • Manage Booking: A landlord views incoming booking requests and updates their status to confirmed, rejected, or completed.
  • Cancel Booking: A student cancels an existing booking with a reason. The system updates status, records the cancellation, and restores room availability.

3.4.3 Functional Requirements

FR-23 The system shall allow authenticated students to create a booking request by specifying a property ID, start date, end date, duration, total amount, and optional notes.
FR-24 The system shall reject booking creation if the target property has zero available rooms.
FR-25 The system shall automatically decrement the property's available room count by one upon successful booking creation and increment it by one upon cancellation or rejection.
FR-26 The system shall assign a default status of "pending" and a payment status of "pending" to newly created bookings.
FR-27 The system shall allow landlords (or administrators) to update booking status to any of the following values: confirmed, rejected, cancelled, or completed.
FR-28 The system shall allow only the booking student to cancel their own booking, recording the cancellation reason, the canceller's ID, and the cancellation timestamp.
FR-29 The system shall allow students to view all their bookings, and landlords to view all bookings associated with their properties, each sorted by creation date descending.

3.5 Review and Rating System

3.5.1 Description and Priority

Priority: Medium. This feature allows authenticated students to submit reviews for properties they have interacted with. Each review includes an overall rating (1–5), sub-category ratings (cleanliness, location, amenities, value, landlord responsiveness), a title, and a text comment. The system enforces a one-review-per-user-per-property constraint and automatically recalculates the property's aggregate rating upon each new review.

3.5.2 Stimulus/Response Sequences

  • Submit Review: A student fills in the review form on a property's detail page. The system checks for existing reviews, persists the new review, recalculates the property's average rating, and displays confirmation.
  • Update/Delete Review: A student modifies or removes their own review. The system verifies ownership and processes the request.

3.5.3 Functional Requirements

FR-30 The system shall allow authenticated users to submit a review for a property consisting of an overall rating (integer 1–5), optional sub-ratings (cleanliness, location, amenities, value, landlord — each 1–5), a title (max 100 characters), and a comment (max 1,000 characters).
FR-31 The system shall enforce a unique constraint on the combination of property and user, preventing duplicate reviews.
FR-32 The system shall recalculate and update the property's average rating and review count each time a new review is created.
FR-33 The system shall allow only the review author (or an administrator) to update or delete a review.
FR-34 The system shall return all reviews for a given property in reverse chronological order, populating the reviewer's name and profile image.

3.6 Dashboard and Analytics

3.6.1 Description and Priority

Priority: Medium. This feature provides authenticated users with a role-specific dashboard interface. Students see their booking history and counts. Landlords see their property portfolio with status summaries and incoming booking requests. The platform also exposes aggregate statistics (total properties, available properties, total bookings) for the landing page.

3.6.2 Stimulus/Response Sequences

  • Access Dashboard: An authenticated user navigates to the dashboard. The system detects their role and renders the appropriate interface with pre-loaded data.
  • View Statistics: The landing page loads aggregate platform statistics via an API call.

3.6.3 Functional Requirements

FR-35 The system shall display a student-specific dashboard showing the user's booking count and a chronologically ordered list of their bookings with property title, dates, status, and amount.
FR-36 The system shall display a landlord-specific dashboard showing the user's property count and a list of their properties with title, location, price, and current status.
FR-37 The system shall provide a public statistics API endpoint returning the total number of properties, the number of available properties, and the aggregate booking count.
FR-38 The system shall restrict dashboard access to authenticated users only, redirecting unauthenticated visitors to the login page.

4. External Interface Requirements

4.1 User Interfaces

The StudentStay platform presents a responsive, browser-based user interface. The following role-specific interfaces are defined:

4.1.1 Public Interface (Unauthenticated Users)

  • Landing Page (/): Hero section with search bar (university/location input and property type selector), platform statistics, featured property grid, "How It Works" step guide, and feature highlights.
  • Properties Page (/properties): Full property catalog with sidebar filter panel (university, city, district, type, price range, amenities, gender) and paginated property card grid.
  • Property Details Page (/property-details): Comprehensive property view with image gallery, description, amenity tags, room details, house rules, pricing, landlord contact information, and user reviews.
  • Registration Page (/register): Multi-field form with role selector (Student/Landlord), conditional student-specific fields (university, student ID), password confirmation, and validation feedback.
  • Login Page (/login): Email and password fields with error messaging and redirect to dashboard.

4.1.2 Student Interface

  • Student Dashboard: Booking count summary, chronological booking list with property title, dates, status, and amount. Quick action links for browsing properties.
  • Booking Form: Date picker, duration, amount, and notes fields accessible from property detail pages.
  • Review Form: Overall rating selector, sub-category ratings, title, and comment fields.

4.1.3 Landlord Interface

  • Landlord Dashboard: Property count summary, property list with title, location, price, and status. "Add New Property" action button. Incoming booking requests with approval/rejection controls.
  • Add Property Page (/add-property): Multi-section form organized into Basic Information, Location, Pricing & Rooms, and Contact Information sections.

All interfaces use the Inter font family, Font Awesome icons, and a consistent color scheme with primary indigo-blue tones. The navigation bar adapts to show authenticated user information (avatar, name, logout) when logged in.

4.2 Hardware Interfaces

StudentStay is a software-only web application and does not interface directly with any specialized hardware. The system requires:

  • A server machine (physical or virtual) capable of running Node.js v14+.
  • Client devices with display screens, standard input peripherals, and network connectivity.
  • No peripheral devices (printers, scanners, biometric readers) are required.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

NFR-01 The system shall respond to standard API requests (property listing, search, booking CRUD) within 500 milliseconds under normal load conditions (up to 100 concurrent users).
NFR-02 The system shall support paginated responses to prevent excessive data transfer, with a default page size of 12 items and a maximum of 20 items per request.
NFR-03 The system shall serve static front-end assets (HTML, CSS, JS) with HTTP compression (gzip) enabled via the compression middleware.
NFR-04 The database shall maintain indexes on frequently queried fields (address.city, university, price, type, property-user composite for reviews) to optimize query performance.

5.2 Security Requirements

NFR-05 The system shall hash all user passwords using bcrypt with a minimum salt rounds value of 10 and shall never store or transmit plaintext passwords.
NFR-06 The system shall authenticate all protected API endpoints by verifying the presence and validity of a Bearer JWT in the Authorization header.
NFR-07 The system shall enforce role-based access control, ensuring that landlord-only operations (property CRUD) and student-only operations (booking creation) are restricted to the appropriate roles.
NFR-08 The system shall apply HTTP security headers via Helmet.js, including X-Content-Type-Options, X-Frame-Options, and Strict-Transport-Security.
NFR-09 The system shall enforce API rate limiting at 100 requests per 15-minute window per IP address to mitigate brute-force and denial-of-service attacks.
NFR-10 The system shall validate and sanitize all user input on the server side using express-validator before processing or persisting data.
NFR-11 The system shall limit JSON request body size to 10 MB to prevent payload-based attacks.

5.3 Software Quality Attributes

Usability

NFR-12 The system shall provide a responsive user interface that renders correctly on viewports ranging from 320px (mobile) to 2560px (desktop), using flexible layouts and media queries.
NFR-13 The system shall display clear, user-friendly validation error messages in real time when form submissions contain invalid data.

Reliability

NFR-14 The system shall gracefully handle MongoDB connection failures by logging warnings and automatically attempting reconnection every 10 seconds, allowing the front-end to remain accessible during database outages.
NFR-15 The system shall return structured JSON error responses for all API errors, including appropriate HTTP status codes and descriptive messages, without exposing internal stack traces in production.

Maintainability

NFR-16 The codebase shall follow a modular MVC-inspired architecture with clear separation between models, controllers, routes, middleware, and views.
NFR-17 The system shall use ES Module syntax (import/export) consistently throughout all server-side code for modern JavaScript compatibility.

Scalability

NFR-18 The system architecture shall support migration from a local MongoDB instance to MongoDB Atlas cloud without code changes beyond the connection string environment variable.

Compatibility

NFR-19 The system shall function correctly on the latest two stable versions of Chrome, Firefox, Edge, and Safari browsers.

Portability

NFR-20 The system shall be deployable on any operating system (Windows, Linux, macOS) that supports Node.js v14 or higher, with all configuration managed through environment variables.

6. Other Requirements

6.1 Introduction

This chapter addresses additional requirements that fall outside the functional and non-functional categories covered in Chapters 3 and 5. These include domain-specific constraints inherent to the student housing context, data management requirements, design constraints imposed by the chosen technology stack, and the testing strategy for validating the delivered system.

6.2 Domain Requirements

DR-01 The system shall default all property pricing to Bangladeshi Taka (BDT) and support billing periods of monthly, semester, or yearly.
DR-02 The system shall validate Bangladeshi mobile phone numbers against the pattern /^(\+88)?01[3-9]\d{8}$/, accepting numbers with or without the +88 country prefix.
DR-03 The system shall support gender-specific housing rules (male-only, female-only, any) to comply with local cultural norms and landlord preferences in Bangladeshi student housing.
DR-04 The system shall associate each property listing with a named university and a numeric distance value (in kilometers) to support proximity-based filtering.
DR-05 The system shall maintain an address hierarchy of street, area, city, and district consistent with Bangladeshi geographic administrative divisions.
DR-06 The system shall support student-specific profile fields (university name, student ID) to differentiate student users from landlords and facilitate future verification workflows.

6.3 Data Requirements

The system manages the following core data categories:

6.4 Design Constraints

  • The system must adhere to the RESTful API design paradigm for all server-client communication.
  • All data persistence must use MongoDB with Mongoose schemas; no additional database systems are permitted in Version 1.0.
  • Authentication must be stateless using JWT; server-side session storage is not used.
  • The front-end must use vanilla HTML/CSS/JavaScript without external UI frameworks (React, Vue, Angular).
  • All server-side code must use ES Module syntax (.mjs extension or "type": "module" in package.json).
  • The application must be configurable entirely via environment variables (.env file).

6.5 Testing Requirements

The following testing areas shall be addressed to validate the StudentStay system:

Functional Testing

  • Authentication Module: Verify registration, login, duplicate email rejection, JWT issuance, profile update, and password change flows.
  • Property Module: Verify CRUD operations, validation enforcement, ownership authorization, and search/filter accuracy.
  • Booking Module: Verify booking creation, availability checks, status transitions, room count adjustments, and cancellation workflows.
  • Review Module: Verify review creation, duplicate prevention, rating recalculation, ownership-based update/delete, and listing retrieval.

Non-Functional Testing

  • Performance: Measure API response times under simulated concurrent user loads.
  • Security: Test for common vulnerabilities (SQL/NoSQL injection via malformed input, unauthorized access to protected routes, JWT tampering, rate limit bypass attempts).
  • Usability: Validate responsive layout across multiple viewport sizes and browsers.
  • Reliability: Test system behavior during database unavailability and network interruptions.

Traceability

Each test case should reference the corresponding requirement identifier (FR-XX, NFR-XX, DR-XX) to maintain full traceability between requirements specification and test coverage. Any defects discovered during testing shall be categorized by severity (Critical, Major, Minor) and linked back to the affected requirement.

A.6.6 SSD 6: Manage Booking (Landlord Approval/Rejection)

This sequence covers the landlord's booking management flow, where the landlord reviews a pending booking and updates its status, with room availability restored upon rejection.

Interaction Flow:
1. Landlord → Browser: navigateToDashboard()
2. Browser → BookingController: GET /api/bookings/landlord-bookings [Bearer token]
3. BookingController → BookingModel: find({landlord: userId}).populate('property', 'student')
4. BookingModel → Database: query → pendingBookings[]
5. BookingController → Browser: 200 {bookings: pendingBookings[]}
6. Browser → Landlord: displayPendingBookings()
7. Landlord → Browser: updateStatus(bookingId, 'confirmed' or 'rejected')
8. Browser → BookingController: PUT /api/bookings/:id/status {status}
9. BookingController → BookingModel: findById(id) → booking
10. BookingController → BookingController: verifyOwnership(booking.landlord === userId)
11. BookingController → BookingModel: save({status: newStatus})
12. [If rejected] BookingController → PropertyModel: update(rooms.available += 1)
13. BookingController → Browser: 200 {success: true, booking}
14. Browser → Landlord: showStatusUpdateConfirmation()
